KRONOS SWARM CORE — Context File
====================================
Last updated: 2026-06-12

OVERVIEW
--------
A simulated AI-driven "swarm intelligence" engine for football match analysis.
Five distinct AI agents analyze in-game telemetry, produce textual verdicts,
and a fracture calculator measures swarm disagreement. Full-stack app:
Python SSE backend + React/TypeScript/Vite/Tailwind frontend.

CURRENT STATE — COMPLETE
------------------------
Backend: 18/18 tests passing. Full SSE server on port 3000.
Frontend: Production-quality React app with War Room command center layout.
LLM Gateway: Provider-based architecture (mock/bob/hybrid) with hybrid fallback.

ARCHITECTURE / DATA FLOW
------------------------
SSE → KronosProvider → normalizeKronosPacket → KronosContext → useKronos() → Components

process_next_tick():
  → KronosMatchTicker.generate_tick() → KronosTelemetryPacket
  → Each of 5 agents constructs prompt → LLMGateway.generate() → LLMResponse
  → SwarmFractureCalculator.calculate(verdicts) → metrics
  → Return { telemetry, debate_outputs, swarm_metrics, provider_metadata }

LLM Gateway Modes:
  mock:   Always MockProvider (deterministic, original behavior)
  bob:    Always BobProvider (failures surface as RuntimeError)
  hybrid: Try BobProvider → on failure → log warning → MockProvider fallback

KEY DECISIONS
-------------
- All logic engines (verdictEngine, eventEngine, swarmNormalizer, swarmCohesion,
  fractureAttribution) are pure functions — no React/browser imports.
- Frontend components read from single source of truth: useKronos() → KronosContext.
- Backend uses _build_telemetry() to flatten nested metric dicts.
- provider_metadata in orchestrator output is optional (frontend doesn't require it).
